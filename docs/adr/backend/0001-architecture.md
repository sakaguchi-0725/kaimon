# ADR-0001: アプリケーションアーキテクチャ

## コンテキスト

Go による REST API の設計において、コードの構造・レイヤー間の依存方向・ドメイン間の連携方法を統一する必要がある。

## 決定

### ディレクトリ構成

集約単位で `internal/` 配下にパッケージを配置する。各パッケージは以下のファイルで構成する。

```
internal/{aggregate}/
├── {domain}.go    # ドメインモデル・値オブジェクト
├── handler.go     # HTTPハンドラ & Service インターフェース
├── service.go     # アプリケーションサービス & Repository インターフェース
├── repository.go  # リポジトリ実装
├── module.go      # DI・ルーティング登録
├── resolver.go    # 他ドメインへの読み取り公開（必要な場合のみ）
└── creator.go     # 他ドメインへの書き込み公開（必要な場合のみ）
```

テンプレートは `internal/_template/` に配置する。

### レイヤー間の依存方向

依存は上から下への一方向とする。

```
handler → service → repository
```

- handler は Service インターフェースに依存する
- service は Repository インターフェースに依存する
- ドメインモデル（{domain}.go）はどのレイヤーにも依存しない

### パッケージ間（ドメイン間）の依存方向

パッケージ間の依存はコンテキストマップに従う。コンテキストマップに明記されていない依存は許可しない。

```
例）コンテキストマップに以下が定義されている場合
  店舗 → 商品
  商品 → 注文

  注文 → 店舗 は NG（コンテキストマップに定義がないため）
```

コンテキストマップは `docs/design/` 配下に別途用意する。

### exported / unexported の使い分け

| 要素 | 公開範囲 | 理由 |
|------|---------|------|
| ドメインモデル（エンティティ・値オブジェクト） | unexported | パッケージ外に漏らさない |
| Service / Repository インターフェース | 型は exported、メソッドは unexported | DI の引数型として利用、パッケージ外からの実装は不可 |
| handler / service / repository 構造体 | unexported | DI・ルーティングは同一パッケージ内（module.go）で完結 |
| リクエスト/レスポンス構造体 | exported | 統合テスト（tests/）から参照するため |
| Resolver / Creator | exported | 他ドメインから利用するため |
| Module | exported | main やルーティング登録で参照するため |

### Module パターン

各パッケージは `Module` 構造体を公開エントリポイントとする。

- `NewModule()` で DI を行い、handler / service / repository を組み立てる
- `RegisterRoutes()` で Echo のルーティングを登録する
- 外部に公開する `Resolver` / `Creator` を保持する（不要な場合は省略）

### service メソッドの入出力

- API と service が 1対1 の場合は handler の request/response 構造体をそのまま受け渡す（レイヤー間の詰め替えは行わない）
- Creator 等、複数から呼び出される場合は request/response とは別の DTO（CreateInput 等）を定義する
- handler にドメインモデルを返却しない。response 構造体に詰め替えて返す

### ドメイン間の連携

- 他ドメインの公開された構造体・型への依存は OK
- 他ドメインのサービス・リポジトリを直接呼び出さない
- 読み取り: Resolver 経由。利用側が自パッケージ内にインターフェースを定義し、参照先の Resolver が暗黙的に満たす
- 書き込み: Creator 経由。Creator は service を経由してドメインロジック（バリデーション等）を共有する
- Creator はほとんどのケースでは発生しない。大量に作成されるなら設計を見直す
- Creator は同期処理である必要がある場合のみ作成する
- メール通知やプッシュ通知といった副作用は Creator ではなくイベント駆動で対応する

### Resolver の公開構造体

- 他ドメインに必要最小限の情報だけ公開する
- 値オブジェクトはプリミティブ型に変換する

### Repository の制約

- リポジトリはドメイン単位でデータの取得・更新を行う
- 特定のフィールドだけ取得するようなメソッドの作成は禁止（アンチパターン）

## 結果

### 良い影響

- パッケージ構成が統一され、新規ドメイン追加時にテンプレートからコピーできる
- unexported を基本とすることで、パッケージ間の不要な結合を構造的に防止できる
- Module パターンにより DI とルーティングの責務が明確になる
- Resolver / Creator による間接参照で、ドメイン間の結合度を低く保てる
- コンテキストマップにより、許可された依存方向が明確になる

### 悪い影響

- パッケージ内のファイル数が多い（最大7ファイル）。小さなドメインでは過剰に感じる可能性がある
- Resolver / Creator の間接参照により、コードのトレーサビリティがやや低下する
