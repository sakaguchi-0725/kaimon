# Git ワークフロー

## コミットメッセージ

Conventional Commits に従う。メッセージは日本語で記述する。

```
<type>: <説明>
```

### type 一覧

| type | 用途 |
|------|------|
| feat | 新機能 |
| fix | バグ修正 |
| refactor | リファクタリング（機能変更なし） |
| test | テストの追加・修正 |
| docs | ドキュメントの変更 |
| chore | ビルド・設定・依存関係の変更 |

### 例

```
feat: 顧客登録APIを実装
fix: メールアドレスの一意性チェックで自身を除外していない問題を修正
refactor: 顧客ドメインのバリデーションロジックを値オブジェクトに移動
test: 顧客削除の受入基準に対応するテストを追加
```

## コミット粒度

- NEVER ビルドが通らない状態でコミットしない
- 実装・テスト・ドキュメントはそれぞれ別コミットにする

```
feat: 顧客登録のドメインモデルを実装
feat: 顧客登録APIを実装
test: 顧客登録の単体テスト・統合テストを追加
docs: 顧客登録のAPI仕様書を追加
```

## ブランチ命名

```
feat/{ドメイン名}-{ユースケースの英語名}
```

例: `feat/customer-create`, `fix/customer-email-uniqueness`

## Pull Request

ALWAYS PR は Draft で作成する。

### テンプレート

`.github/pull_request_template.md` のテンプレートに従う。各セクションの記載内容:

- **対応したこと**: PRに含まれる変更の概要を箇条書きで簡潔に。差分やコメントを見ればわかる詳細は書かない
- **対応していないこと**: スコープ外とした内容や、既知の制限事項
- **関連Issue**: AskUserQuestion で「関連する Issue 番号はありますか？」と「この PR で close してよいですか？」を確認する。close する場合は `- close: #123`、参照のみの場合は `- #123` とする
- **補足**: AIレビュー結果のサマリー（指摘件数と対応状況）、設計判断の背景など

### PR のインラインコメント

PR 作成後、差分の該当行に実装意図をインラインコメントとして投稿する。レビュアーがコードを読む量を減らすことが目的。

#### コメントすべき箇所

- ビジネスルールの実装理由（「なぜこの条件分岐があるか」）
- 設計判断（「なぜ resolver 経由にしたか」「なぜこの構造にしたか」）
- 業務ルール.md やドメインモデル.md の該当箇所への参照
- 複雑なクエリや変換ロジックの意図

#### コメントしなくてよい箇所

- 読めば自明なコード（コンストラクタ、単純な getter 等）
- ルール通りの定型実装（フォーマット、エラーハンドリング等）

#### 投稿方法

```bash
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments \
  -f body="{コメント本文}" \
  -f path="{ファイルパス}" \
  -F line={行番号} \
  -f commit_id="$(git rev-parse HEAD)"
```
