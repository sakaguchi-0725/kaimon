# 設計ドキュメント生成 Skill

`docs/diagrams/` 配下の drawio 図（ドメインモデル図・ユースケース図・オブジェクト図）を読み、**drawio から読み取れない細かい仕様（バリデーション・状態遷移・エッジケース・受入基準など）を AskUserQuestion で磨き込みながら**、指定ドメインの設計ドキュメント（`index.md` / `ドメインモデル.md` / `{ユースケース}.md`）の叩きを生成する。

## When to Activate

- 「設計ドキュメントを作って」「ドメインモデルを markdown に落として」と依頼されたとき
- sudoモデリングの drawio 成果物が揃い、実装に向けた設計ドキュメントが必要になったとき
- 新規ドメインを追加したとき、または既存ドメインの設計を大きく変更したとき

## 本 Skill の基本方針

- ALWAYS drawio を**情報源の起点**として扱い、そこから読み取れる事実（エンティティ・属性・関係・ユースケース一覧など）はそのまま反映する
- ALWAYS drawio から読み取れない仕様（バリデーション・境界値・状態遷移の完全性・受入基準など）は **AskUserQuestion でユーザーに質問し、回答を叩きとして反映する**
- NEVER drawio に無いからという理由だけで `<!-- TODO -->` や `{...}` プレースホルダで埋めて終わらせない。質問 → 回答 → 反映のサイクルを回す
- NEVER 推測や常識で補完した内容を確定事項として書かない。必ず AskUserQuestion を経由して確定させる
- ALWAYS 質問しても確定しなかった項目のみ、**検討メモ**セクションに残す（本文には書かない）

## 入力

- ドメイン名（例: `family`, `account`, `shopping-list`）
- 指定がない場合: AskUserQuestion で対象ドメインを選ばせる

## 参照ファイル

| ファイル | 用途 |
|---------|------|
| `docs/diagrams/domain-model.drawio` | エンティティ・集約・ValueObject・Enum・関係・付箋（業務ルールの種）を読み取る |
| `docs/diagrams/usecase.drawio` | ユースケース一覧・アクター・ユースケース間の関係を読み取る |
| `docs/diagrams/object.drawio` | インスタンスレベルの具体例・状態遷移（受入基準の元ネタ）を読み取る |
| `docs/diagrams/context-map.drawio` | 他ドメイン・外部サービスとの依存方向（U/D）を読み取る |
| `docs/design/_template/index.md` | `index.md` 生成時のテンプレート |
| `docs/design/_template/ドメインモデル.md` | `ドメインモデル.md` 生成時のテンプレート |
| `docs/design/_template/ユースケース.md` | `{ユースケース}.md` 生成時のテンプレート（1 ユースケース = 1 ファイル） |

### テンプレートと出力の対応

| 出力ファイル | 使用するテンプレート | 生成単位 |
|------------|------------------|--------|
| `docs/design/{domain}/index.md` | `_template/index.md` | ドメインごとに 1 ファイル |
| `docs/design/{domain}/ドメインモデル.md` | `_template/ドメインモデル.md` | ドメインごとに 1 ファイル |
| `docs/design/{domain}/{ユースケース}.md` | `_template/ユースケース.md` | ユースケースごとに 1 ファイル |

## drawio XML の読み方

drawio ファイルは XML 形式で記述されている。設計情報を読み取る際の要点を以下に示す。

### 基本構造

```xml
<mxfile>
  <diagram id="..." name="...">
    <mxGraphModel>
      <root>
        <mxCell id="0"/>
        <mxCell id="1" parent="0"/>
        <!-- 以降が実際の要素 -->
        <mxCell id="{id}" value="{表示テキスト}" style="..." vertex="1" parent="1">
          <mxGeometry ... />
        </mxCell>
      </root>
    </mxGraphModel>
  </diagram>
</mxfile>
```

### 要素の識別

drawio の `style` 属性と `value` の HTML から要素種別を判別する。

| 要素の種別 | 識別の手がかり |
|-----------|--------------|
| 集約境界（フレーム） | `style` に `dashed=1` + `fillColor=none`、`value` に「〇〇集約」を含む |
| エンティティ / VO | `style` に `rounded=1`、`value` に `<b>エンティティ名</b>` + 属性リスト |
| 付箋（Enum / VO / 業務ルール補足） | `style` に `shape=note` |
| エッジ（関係線） | `edge="1"`、`source` と `target` の ID が対応するセルを指す |
| 多重度ラベル | `parent` が エッジの ID である `edgeLabel` スタイルのセル |
| ユースケース楕円 | `style` に `ellipse`、`value` にユースケース名 |
| アクター | `style` に `shape=mxgraph.flowchart.start_2` または人型アイコン系 |
| レビュー指摘の付箋 | `fillColor=#FFF2CC` + `[CRITICAL]` / `[HIGH]` 等のプレフィックス |

### value の HTML タグ除去

`value` には HTML タグが含まれることがある。テキスト抽出時は `<b>`, `<div>`, `<br>` 等を除去してプレーンテキストとして扱う。

```
// PASS: HTML タグを除去してエンティティ名・属性を抽出する
value = "<b>家族メンバー</b><div>id</div><div>accountId</div><div>role</div><div>joinedAt</div>"
→ エンティティ名: 家族メンバー
→ 属性: id, accountId, role, joinedAt

// FAIL: HTML タグを含んだままドキュメントに貼り付ける
→ ドキュメントが HTML タグで汚染され可読性が損なわれる
```

### ドメインのフィルタリング

ドメインモデル図には複数の集約が描かれる。引数で指定されたドメインに対応する集約フレームの配下にある要素のみを抽出する。フレームと要素の対応は `parent` 属性で判定するのではなく、座標（`mxGeometry` の `x`, `y`, `width`, `height`）による包含関係で判定する。フレームの矩形領域内に中心点が収まる要素を配下とみなす。

## 手順

本 Skill は「drawio 抽出 → ギャップ分析 → リファインメント質問 → 反映 → 生成」の流れで叩き md を作る。

### Step 1: ドメイン名の確認

引数でドメイン名が指定されている場合はそのまま進む。未指定の場合は AskUserQuestion を使用する。

### Step 2: drawio ファイルの読み込み

以下のファイルを存在確認してから読む。ファイルが 1 件も見つからない場合は処理を中断し、先に drawio 図を作成するよう案内する。

- `docs/diagrams/domain-model.drawio`
- `docs/diagrams/usecase.drawio`
- `docs/diagrams/object.drawio`
- `docs/diagrams/context-map.drawio`

### Step 3: drawio から一次情報を抽出

指定ドメインに対応するフレーム配下から以下を抽出する。ここで抽出する情報は「事実」として扱う。

- 集約名・集約ルート・エンティティ・ValueObject・Enum・属性・関係（多重度含む）
- ユースケース名・アクター・ユースケース間の関係
- オブジェクト図のインスタンス例（受入基準の素材）
- コンテキストマップ上の他ドメイン / 外部サービス依存
- 付箋の業務ルール（例:「オーナーのみ招待可能」「購入済みは物理削除しない」）
- drawio 上のレビュー指摘付箋（`[CRITICAL]` / `[HIGH]`）→ **Step 4 の質問候補の優先材料**

### Step 4: 固定観点でギャップ分析（drawio から読み取れない箇所を洗い出す）

テンプレート（`_template/index.md` / `ドメインモデル.md` / `ユースケース.md`）を Read し、テンプレートの各項目のうち **drawio から埋められない項目** を列挙する。観点は `review-business-rules` に揃える。

#### 観点A: 網羅性
- 各エンティティ・VO に対応する業務ルールが drawio 付箋に書かれているか
- 全ユースケースがドメインに割り当てられているか（ユースケース図のグルーピングと引数ドメインの整合）
- ステータスごとに「何ができて、何ができないか」が読み取れるか

#### 観点B: 明確さ
- 付箋に曖昧表現（「基本的に」「〜の場合がある」「適宜」「など」）が含まれていないか
- エンティティの属性名から型・必須/任意が判別できるか（不明なら質問対象）

#### 観点C: 境界値・エッジケース
- 文字数上限／数値の上下限／ゼロ値／空文字／null の扱いが drawio に定義されているか
- 一意性制約（email 等）の正規化ルールが定義されているか

#### 観点D: 状態遷移の完全性
- Enum 付箋（状態）に対して、全ての正当な遷移パスがエッジ／付箋で表現されているか
- 不正遷移（例: 購入済み → カート戻し）の扱いが定義されているか
- 自動遷移のトリガー（時間・アクション）が定義されているか

#### 観点E: ドメイン間整合性
- 他集約への参照が **ID 参照か・スナップショットか**、コンテキストマップと一致するか
- 外部サービス連携の責務境界（interface を誰が定義するか）が読み取れるか

#### 観点F: 受入基準の具体化
- オブジェクト図の具体例から Given/When/Then に落とせるか
- 正常系・異常系それぞれに対応する具体例があるか

### Step 5: リファインメント質問（AskUserQuestion）

Step 4 で洗い出したギャップのうち、**重要度の高いものから** AskUserQuestion で質問する。

#### 質問ルール

- ALWAYS `AskUserQuestion` を使う（CLAUDE.md 準拠）
- ALWAYS 1 回のリクエストで最大 **4 問まで**。重要度順（CRITICAL → HIGH → MEDIUM）で出す
- ALWAYS 選択肢には **推奨案 + 代替案** を並べ、推奨案のラベル末尾に `(推奨)` を付ける
- ALWAYS 選択肢の `description` に**選んだ場合の影響**を明記する（例: 「DB に email 正規化カラムを追加」「Cognito 側の email_verified と突合」）
- NEVER 同一カテゴリの質問を分散させない（境界値なら境界値でまとめる）
- NEVER 答えが「はい / いいえ」だけで閉じる質問をしない（選んだ後の仕様に分岐があることを示す）

#### 質問のカテゴリ例

| カテゴリ | 具体例 |
|---------|------|
| バリデーション | 「家族名の文字数上限は？」「email 正規化は完全一致 / 小文字化 / 全角半角寄せ のどれ？」 |
| 境界値 | 「quantity = 0 は許容する？」「招待の有効期限切れ後の再招待は可能？」 |
| 状態遷移 | 「購入済み → 未購入 の差し戻しは可能？」「キャンセル状態の再開は？」 |
| 受入基準の代表シナリオ | 「オブジェクト図ケース1 / ケース2 / ケース3 のどれを正常系の代表例にする？」 |
| 外部サービス境界 | 「Cognito サインアップはこの API が呼ぶ / フロントから直接呼ばせる のどちら？」 |
| ドメイン間整合 | 「ユーザー⇔アカウントの多重度は 1:1 / 1:0..1 のどちら？」 |

#### 質問回数の上限

- 1 ドメインあたり AskUserQuestion は最大 **3 ラウンド** まで（合計 12 問が上限）
- 3 ラウンドで確定しきらなかった項目は **検討メモ**セクションに残し、本文には確定した分だけ書く
- 複数ユースケースがある場合、ユースケース共通事項 → ユースケース個別事項 の順で進める

### Step 6: 出力ファイルの確認と上書き判断

出力先ディレクトリを確認する。

```
docs/design/{domain}/index.md
docs/design/{domain}/ドメインモデル.md
docs/design/{domain}/{ユースケース}.md  # ユースケース数分
```

既存ファイルが存在する場合は AskUserQuestion で上書きの可否を確認する。

### Step 7: ファイルの生成（drawio + 回答を統合）

テンプレートを Read し、プレースホルダを以下のルールで置換する。

#### 置換ルール

| プレースホルダ | 埋め方 |
|-------------|------|
| エンティティ名・属性・関係 | drawio の抽出結果（一次情報） |
| 集約の操作ルール・業務ルール | drawio 付箋 + Step 5 の回答 |
| 共通バリデーション（文字数・email 正規化等） | Step 5 の回答 |
| ライフサイクル（状態遷移表・遷移ルール） | drawio の Enum 付箋 + Step 5 の回答 |
| 受入基準（Given/When/Then） | オブジェクト図の具体例 + Step 5 の回答 |
| `{実際のpathを入力}`（API 仕様書リンク） | `docs/api/` 配下に対応ファイルがあれば置換、無ければプレースホルダのまま |

#### 受入基準の書き方

`review-business-rules` と同じフォーマットに揃える。オブジェクト図の具体例（例: `acc-001 サトウ太郎 / fg-001 サトウ家`）をそのまま使うと具体性が上がる。

```markdown
### {ユースケース名}

**正常系: {シナリオ名}**
- Given: {前提条件（オブジェクト図の具体例を使う）}
- When: {操作}
- Then: {期待結果}

**異常系: {シナリオ名}**
- Given: {前提条件}
- When: {不正な操作}
- Then: エラー（{エラーメッセージ}）
```

#### 検討メモの扱い

Step 5 の質問で確定しきらなかった項目は、各 md の末尾に以下のセクションを追加して残す。本文には書かない。

```markdown
## 検討メモ（未確定事項）

- {論点}: {現時点の選択肢と未確定の理由}
```

#### テンプレート流し込みの原則

- ALWAYS テンプレートファイルを実行時に Read で読み込む（本 SKILL 内にはテンプレート本文を転記しない）
- ALWAYS テンプレートの見出し構成・テーブル列・本文構造は変更しない
- ALWAYS drawio から読み取れた情報 + Step 5 の回答で本文を埋める
- NEVER 回答を得ていない箇所を推測で埋める
- NEVER 本文に `<!-- TODO -->` や長文の NOTE コメントをばらまかない。確定事項は本文、未確定事項は「検討メモ」セクションに集約する

#### ユースケースファイルの命名

- ファイル名は日本語ユースケース名をそのまま使う（例: `家族を作成する.md`）
- 既存の設計ドキュメントで英語ファイル名が使われている場合は、そのドメインの慣習に合わせる

#### index.md 生成時の注意

`_template/index.md` の「関連ドキュメント」セクションには単一ファイル前提の `[ユースケース](ユースケース.md)` リンクが入っている。1 ユースケース = 1 ファイルに分割するため、以下のとおり置換する。

- 単一ファイル前提の行は削除し、「ユースケース」セクションに各 `{ユースケース}.md` へのリンクを列挙する
- `{実際のpathを入力}` は対応ファイルがあれば置換、無ければプレースホルダを残す

#### {ユースケース}.md 生成時の注意

- **スコープ / ライフサイクル / 共通バリデーション**: ドメイン全体に共通するため、各 `{ユースケース}.md` に同じ内容を重複記載する
- **ユースケース名セクション**: そのファイルが対象とする 1 ユースケースのビジネスルール
- **受入基準**: そのユースケースの受入基準のみ

### Step 8: 生成結果の報告

生成したファイル一覧、リファインメントで確定した主要仕様、未確定事項（検討メモ）を整理して報告する。

```
## 生成結果

- 対象ドメイン: {domain}
- 生成ファイル:
  - docs/design/{domain}/index.md
  - docs/design/{domain}/ドメインモデル.md
  - docs/design/{domain}/{ユースケース1}.md
  - ...

## リファインメントで確定した仕様

- {カテゴリ}: {確定内容}

## 未確定事項（検討メモに記載）

- {論点}: {未確定の理由}
```

## PASS / FAIL コード例

### ギャップ分析と質問

```
// PASS: drawio 不足箇所を観点別に洗い出し、AskUserQuestion で埋める
→ 観点C（境界値）: email 正規化ルールが drawio に未記載
→ AskUserQuestion で「完全一致 / 小文字化 / 全角半角寄せ」を選ばせる
→ 回答を共通バリデーションセクションの本文に反映

// FAIL: drawio に無いからと本文を空欄 / プレースホルダで残す
→ 叩きとして使えない。受入基準が書けない
```

### エンティティの属性抽出

```
// PASS: HTML タグを除去し、属性を箇条書きで整理する
value = "<b>招待</b><div>id</div><div>invitedBy</div><div>inviteeEmail</div><div>status</div>"
→ エンティティ名: 招待
→ 属性: id, invitedBy, inviteeEmail, status

// FAIL: value をそのままドキュメントに貼り付ける
→ "<b>招待</b>..." が混入する
```

### 受入基準の書き方

```
// PASS: オブジェクト図の具体例を使って Given/When/Then を書く
**正常系: オーナーが招待を発行する**
- Given: サトウ家(fg-001) にオーナー サトウ太郎(mb-001) が所属している
- When: サトウ太郎が jiro@example.com 宛に招待を発行する
- Then: 招待(status=未承諾, expiresAt=invitedAt+7日) が作成される

// FAIL: Given/When/Then を抽象名だけで書く
- Given: {前提条件} / When: {操作} / Then: {期待結果}
→ 叩きとして使えない
```

## エラーハンドリング

| エラーケース | 対処 |
|------------|------|
| `docs/diagrams/domain-model.drawio` が存在しない | 処理を中断し、先に drawio 図を作成するよう案内する |
| `docs/diagrams/usecase.drawio` が存在しない | ドメインモデル.md と index.md のみ生成する旨を報告し、ユースケースファイルは生成しない |
| 指定ドメインに対応する集約フレームが見つからない | AskUserQuestion でドメイン名の表記（日本語 / 英語 / 略称）を確認し、手動で対応フレームを特定する |
| drawio の XML が解析できない（構文エラー等） | エラー箇所を報告し、ファイルの修正を求める |
| ユースケース図にドメイン境界が明示されていない | ユースケースを列挙して AskUserQuestion で所属ドメインを選ばせる |
| 既存ファイルが存在する | AskUserQuestion で上書き確認 |
| ドメインモデル図に当該集約が欠落している（例: ユーザー集約が未描画） | `context-map.drawio` / オブジェクト図 / ユーザー回答を統合して叩きを作る。本文冒頭に WARNING コメントは書かず、「検討メモ」に「ドメインモデル図へ反映が必要」と残す |

## アンチパターン

```
// FAIL: drawio を機械的にmdに写すだけ
→ バリデーション・境界値・受入基準が空欄の叩きになり、実装に入れない
→ 固定観点（Step 4）で不足を洗い出し、AskUserQuestion で埋める

// FAIL: 本文中に TODO / NOTE / WARNING コメントを散りばめる
→ 読みづらく、確定事項と未確定事項が混在する
→ 確定事項は本文へ、未確定事項は末尾の「検討メモ」セクションへ集約

// FAIL: 他ドメインのエンティティを含む
→ 他ドメインへの参照は「集約ルート ID の参照」として表現し、詳細は当該ドメインのドキュメントに任せる

// FAIL: 推測で補完する
→ 必ず AskUserQuestion を経由して確定させる

// FAIL: ユースケースファイルを 1 ファイルにまとめる
→ ユースケース単位で分割する

// FAIL: テンプレートの見出し・テーブル構造を変更する
→ 値のプレースホルダ置換のみ行う

// FAIL: SKILL.md 内にテンプレート本文を転記して使用する
→ 実行時に Read で `_template/*.md` を読み込む
```

## 関連スキル

| スキル | 関係 | 説明 |
|--------|------|------|
| `review-business-rules` | 参考 | 本 Skill の Step 4〜5（固定観点レビュー + リファインメント質問）の観点セットに揃えている |
| `generate-er` | 下流 | 本 Skill が生成する `ドメインモデル.md` を入力として ER 図（DBML）を生成する |
| `review-design` | 上流 | drawio 図そのものをレビューし付箋を追記する。本 Skill の前工程 |
| `drawio` | 上流 | drawio 図を作成・更新する。本 Skill の前工程 |

### 典型的な実行順序

```
[設計フェーズ]
drawio Skill（図の作成）
  → review-design Skill（図のレビュー・付箋追記）
  → [ユーザーが図を修正]

[ドキュメント化フェーズ]
generate-design-docs Skill（本 Skill: drawio + リファインメント質問 → 叩き md 生成） ← ここ
  → review-business-rules Skill（生成後の追加リファインメント）
  → generate-er Skill（ER 図生成）
```

## 成功指標

- 指定ドメインの全エンティティ・全ユースケースが叩き md に反映されている
- バリデーション・境界値・状態遷移・受入基準の各セクションが **drawio + AskUserQuestion の回答** で埋まっている（空欄・TODO コメントが本文に残らない）
- 未確定事項は「検討メモ」セクションに集約されている
- 他ドメインのエンティティが混在していない
- `index.md` の関連ドキュメントリンクが生成ファイルを正しく参照している

---

**Remember**: 本 Skill のゴールは「drawio の書き写し」ではなく「実装に着手できる叩き md の生成」。drawio で足りない仕様は AskUserQuestion で必ず埋める。推測で書かず、質問で確定させ、それでも残る論点は「検討メモ」に集約する。
